Section 4 of 5: Holly API

How Holly (Openclaw) communicates with the system

Auth Flow

Ian (web browser)
Logs in via Google OAuth or email/password. NextAuth.js session cookie. All /api/v1/* routes require valid session.
Holly (Openclaw AI)
Uses a long-lived API key: X-Holly-API-Key: hky_.... Key is generated in Settings, stored hashed in DB. Separate from Ian's session. All /api/holly/v1/* routes require this header.
Rate limiting: 1000 requests/minute on Holly API routes via Redis sliding window. Returns 429 with Retry-After header.
Real-time (SSE): Holly can subscribe to GET /api/holly/v1/stream for live events (new interactions, task updates, follow-up triggers).

Endpoints

Interactions
POST /api/holly/v1/interactions log a call/event
GET /api/holly/v1/interactions list recent
Contacts
GET /api/holly/v1/contacts search/list
GET /api/holly/v1/contacts/:id full profile
Briefings & Follow-ups
GET /api/holly/v1/briefing morning summary
GET /api/holly/v1/follow-ups pending follow-ups
Tasks & Actions
POST /api/holly/v1/action-items create action
PATCH /api/holly/v1/action-items/:id update status
Projects
GET /api/holly/v1/projects project dashboard
Real-time
SSE /api/holly/v1/stream live event stream
Does this cover what Holly needs? All endpoints return JSON. The briefing endpoint aggregates overdue contacts, pending follow-ups, and active project tasks into a single payload Holly can read each morning.
