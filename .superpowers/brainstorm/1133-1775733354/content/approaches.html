Holly PRM - Architecture Approach

Three ways to structure the system. Pick the one that feels right.

A

Full-stack Next.js Recommended

One codebase, one container on Coolify. React frontend + API routes as the backend. PWA support built-in. TypeScript throughout.

Pros

  • Single repo, single deploy
  • Shared types between UI and API
  • Excellent PWA + offline support
  • Fast to build, easy to iterate

Cons

  • API routes less structured than NestJS
  • Holly API and web UI share same process
  • Less enterprise-grade structure
Stack: Next.js 15 + TypeScript + Prisma + PostgreSQL + Redis
B

NestJS Backend + React Frontend

Two separate services as specified in your tech doc. NestJS handles all API and Holly integration. React (Vite) is the PWA frontend.

Pros

  • Matches your existing spec exactly
  • Strong separation of concerns
  • NestJS ideal for complex API structure
  • Backend can scale independently

Cons

  • Two containers to manage on Coolify
  • More setup overhead for a personal tool
  • Shared types require monorepo tooling
Stack: NestJS + TypeScript + React (Vite) + Prisma + PostgreSQL + Redis
C

Hono Backend + React Frontend

Lightweight, modern alternative to NestJS. Fast HTTP framework with TypeScript-first design. Same two-service split but much less boilerplate.

Pros

  • Very fast, minimal overhead
  • Simpler than NestJS to understand
  • TypeScript-first, great DX
  • Easy to add structure as needed

Cons

  • Less convention than NestJS
  • Fewer built-in enterprise features
  • Still two containers to manage
Stack: Hono + TypeScript + React (Vite) + Prisma + PostgreSQL + Redis
My recommendation: A (Next.js) - For a personal tool with one user, the simplicity of one codebase and one deploy outweighs the structural benefits of NestJS. You can still organise API routes cleanly. Holly's API gets its own route group with API key auth. Prisma gives you type-safe DB access throughout.
